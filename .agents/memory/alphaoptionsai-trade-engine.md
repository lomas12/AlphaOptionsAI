---
name: AlphaOptionsAI trade engine design
description: Architecture decisions for the AlphaOptionsAI Discord options bot's V3 provider abstraction, single-trade decision engine, backtester methodology, and self-learning confidence.
---

The bot (in `discord-bot/`) picks exactly one trade (BUY CALL / BUY PUT / NO TRADE) per `/scan`, never both a call and a put. Confidence blends two halves 50/50: a liquidity/greeks score (OI, volume, vol/OI, IV, spread, distance from spot, DTE) and a "conviction" score built from ~34 tagged technical/options/market/news/sentiment signals (EMA stack, RSI, MACD, ADX, SuperTrend, Ichimoku, relative strength vs SPY, VIX regime, insider activity, unusual-options heuristic, analyst actions, earnings proximity).

**Why tags, not a fixed formula:** each active signal is a named tag with a learned weight in SQLite (`strategy_weights` table, bounded 0.5–2.0, ±5% per win/loss). The 15-minute monitor loop closes any open recommendation that hits TP/SL/expiration and calls `apply_learning()` on that trade's tags — future confidence for the same tag combo shifts from real outcomes, not a static rulebook.

**Provider abstraction (added in V3):** `apis/base.py` defines shared dataclasses (Quote, HistoryResult, OptionChainResult, NewsItem, etc.); every provider module (`yahoo`, `polygon`, `tradier`, `alpaca`, `finnhub`, `benzinga`) implements identical function signatures. `apis/router.py` tries them in priority order and returns the first real result plus which provider served it — this is how "premium data unavailable" is reported honestly instead of fabricated, and how a paid API key can be dropped in later (just set its env var) with zero code changes to callers.

**Action threshold:** confidence must be ≥70% for BUY CALL/PUT, else NO TRADE. The 5-minute auto-scanner (all 18 watchlist tickers) posts to `#trade-alerts` only at ≥85% confidence, liquidity ≥60, and risk/reward ≥2:1 — if no channel named `trade-alerts` exists in any joined guild, it logs a warning and skips posting rather than erroring.

**Trade-plan risk/reward gotcha:** the TP1/TP2/stop multipliers must be chosen so a *typical* trade clears whatever minimum risk/reward the auto-scanner requires — a naive TP1=1.30x/stop=0.55x plan only yields 0.67:1 R:R, which silently fails a 2:1 gate on every trade. Verify the resulting R:R algebraically (or empirically) whenever these multipliers change.

**Data fallbacks, never fabricate:** every factor lacking a free data source is explicit about it rather than guessed — IV rank/percentile is a realized-vol proxy (Yahoo has no historical IV), unusual-options-activity is a labeled heuristic (volume/OI ratio, no real tape data), market breadth is reported as unavailable (needs a premium feed).

**Backtester methodology:** real historical options quotes aren't free, so it backtests a real EMA20/50 crossover on real historical stock prices (Yahoo) and prices the hypothetical option leg via Black-Scholes seeded with trailing realized volatility — always presented as modeled premiums via an explicit methodology note, never as real historical fills. Flags `production_ready` only at 20+ trades.

**yfinance log noise:** yfinance prints "HTTP Error 404 ... No fundamentals data found" directly whenever a symbol has no earnings/fundamentals (expected for ETFs like SPY/QQQ/IWM) even though the caller catches the exception fine. Set `logging.getLogger("yfinance").setLevel(logging.CRITICAL)` to silence it — `ERROR` is not enough since ERROR-level messages still pass an ERROR threshold.

**Account balance / position sizing:** no persistent per-user balance; assumes a default $10,000 account (`ACCOUNT_BALANCE_DEFAULT` in `core/risk.py`) for 2% risk sizing. Per-user balances would need a new command/table.

**Price verification gate (core/market_data.py):** every recommendation must go through `get_verified_quote()` before any calculation — it cross-checks the provider's snapshot price against a second, independently-timestamped reading (Yahoo: latest 1-minute bar), rejects quotes with no timestamp, rejects quotes stale >15min during market hours (weekday 9:30-16:00 ET; outside market hours an old timestamp is expected and fine), rejects >1.5% disagreement between the two readings, and retries once before raising `MarketDataUnavailableError`. Callers must surface that as "Market data unavailable", never fall back to an estimate. **Why:** a single unverified snapshot price can be cached/stale/wrong, and letting a bad price flow into strike selection, greeks, and expected-move math silently corrupts the whole recommendation.

**Yahoo quotes must include extended-hours prices:** `yfinance`'s `fast_info`/`.info` regular-market fields freeze at the prior regular-session close once the market closes -- they don't include after-hours/pre-market trades. Real brokers (Wealthsimple, etc.) show the after-hours price as "current," so a bot doing the same must too, or it looks broken/wrong even though the regular-close price it's showing is technically accurate. Fix: pull `regularMarketPrice`/`postMarketPrice`/`preMarketPrice` each with their own Unix timestamp from `.info`, plus an intraday history call with `prepost=True`, and always take whichever reading has the latest timestamp -- never default to the regular-session field.

**IV Rank/Percentile must be clamped to 0-100%:** comparing current *implied* vol against a trailing *realized*-vol range (the free-data proxy used here) can legitimately push the raw ratio outside 0-100% -- IV usually carries a premium over realized vol and can spike hard ahead of catalysts like earnings. Always clamp the result; an unclamped value like "118%" looks like a data bug even when the underlying price/inputs are correct.

**V4 decision engine (`core/scoring.py`):** replaced the 50/50 liquidity+conviction blend with a category-by-category weighted 0-100 score (Trend/Momentum/Volume/Market/News each budgeted, Options Data scored from the chosen contract/chain, Risk as a penalty/bonus layer folding in earnings proximity, contract risk/liquidity, and the side-matched backtest win rate). `core/options.py.select_best_contract()` runs a Smart Contract Filter (OI/volume/spread/liquidity/theta-decay/pre-earnings-IV thresholds) and ranks by a composite score, auto-falling through to the next-best contract on rejection. `core/backtester.run_backtest()` takes an optional `side` filter so historical stats match the specific call/put being recommended, not a blended average.

**Weighted-bucket scoring denominator pitfall:** when a category score is `clamp(raw / max_possible, -1, 1) * budget` with `raw` built from mutually-exclusive favorable/headwind tag pairs (e.g. "trend aligned" vs "trend against" -- only one can ever be active at once), `max_possible` must be `max(sum(favorable weights), sum(headwind weights))`, NOT their sum. Summing both makes the full budget mathematically unreachable even in a textbook-perfect setup, silently capping every score at ~50-65% of its stated max. Same applies to side-paired tag variants (a "call" tag list can never contain the "bearish" variant of a signal) -- resolve favorable/headwind sets per side rather than using one flat set covering both directions. Verify by feeding an all-favorable synthetic input and confirming the category (and total) can actually hit its max.

**Discord bot workflow restarts and Cloudflare rate limiting:** restarting the bot's workflow repeatedly in a short window can trigger `discord.errors.HTTPException: 429` with a Cloudflare error-1015 HTML page ("you are being rate limited/temporarily banned") on the login call itself -- this is an IP-level, time-based ban unrelated to code correctness. Don't treat it as a bug to fix by restarting again immediately; that extends the ban. Finish and smoke-test all code changes via direct Python calls first, restart once, and if it recurs, wait rather than retry the restart repeatedly.
